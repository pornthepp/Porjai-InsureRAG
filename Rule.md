# Rule.md

## รูปแบบผู้ช่วยสนทนาแบบเมนูนำทาง

เอกสารนี้เป็นกติกาการออกแบบผู้ช่วยประกันภัยของโปรเจกต์ โดยสกัดจากพฤติกรรมการใช้งานที่ผู้ใช้สังเกตจากผู้ช่วยบริการลูกค้าแบบเมนูนำทาง ไม่ใช่เอกสารภายในหรือข้อกำหนดอย่างเป็นทางการของ True

## เป้าหมาย

- ให้ระบบเริ่มบทสนทนาและชี้ทางก่อน ไม่ปล่อยหน้าจอว่างให้ผู้ใช้เดาเอง
- ลดคำถามกำกวม ลดการเรียก LLM และพาผู้ใช้ไปถึงคำตอบเร็ว
- ใช้เมนูเป็นตัวช่วย แต่ไม่ล็อกเส้นทาง ผู้ใช้ต้องพิมพ์คำถามหรือเปลี่ยนหมวดได้ตลอด
- ใช้ RAG และ LLM เฉพาะเมื่อกฎ offline หรือเมนูตอบไม่ได้

## ลำดับการเริ่มสนทนา

1. ระบบทักทายก่อนด้วยภาษาสุภาพ เป็นมิตร และกระชับ
2. ถ้ามีโปรโมชั่น ต้องแสดงเฉพาะข้อมูลที่มีแหล่งข้อมูลปัจจุบัน ห้ามแต่งโปรโมชั่น
3. เชิญให้ผู้ใช้พิมพ์คำถามเองได้
4. แสดงเมนูหลักในกล่องข้อความของผู้ช่วย

เมนูหลักของโปรเจกต์:

- ประกันรถยนต์
- ประกันชีวิต
- การเคลม/แจ้งเหตุ
- โปรโมชั่น
- บริการอื่นๆ
- ติดต่อเจ้าหน้าที่

## รูปแบบเมนูหลายชั้น

- เมื่อผู้ใช้เลือกเมนู ระบบตอบรับก่อน เช่น “ได้ค่ะ” หรือ “ไม่ต้องกังวลนะคะ”
- บอกให้ชัดว่าระบบกำลังช่วยเรื่องอะไร
- ถามคำถามถัดไปเพียงเรื่องเดียว
- แสดงปุ่มตัวเลือกที่สั้นและชัด
- ทุกชั้นควรมี “บริการอื่นๆ” หรือ “กลับเมนูหลัก”
- ปุ่มและคำแนะนำต้องอยู่ในกล่องข้อความของ assistant เพื่อให้หน้าตาเป็นบทสนทนาเดียวกัน

ตัวอย่าง:

1. ผู้ใช้เลือก “ประกันรถยนต์”
2. ระบบถามว่าต้องการดู “ความคุ้มครอง”, “เบี้ยประกัน”, “การเคลม”, “โปรโมชั่น” หรือ “บริการอื่นๆ”
3. ผู้ใช้เลือกหัวข้อ หรือพิมพ์คำถามใหม่เอง

## การเปลี่ยนเส้นทาง

- ผู้ใช้ไม่จำเป็นต้องกดปุ่มตามลำดับ
- ถ้ากดหมวดใหม่ ให้หมวดใหม่ชนะบริบทเดิมทันที
- ถ้าพิมพ์คำถามที่ระบุหมวดชัด ให้หมวดในข้อความล่าสุดชนะหมวดเดิม
- ถ้าพิมพ์คำถามสั้นและกำกวม เช่น “ราคาเท่าไร” ให้ใช้หมวดล่าสุด
- ถ้าเลือก “บริการอื่นๆ” ให้แสดงตัวเลือกใหม่โดยไม่บังคับย้อนเส้นทางเดิม
- ถ้าเลือก “กลับเมนูหลัก” ให้ล้างสถานะเมนูย่อย แต่เก็บประวัติแชท

ลำดับความสำคัญของบริบท:

1. หมวดที่ระบุชัดในข้อความล่าสุด
2. ปุ่มที่ผู้ใช้เพิ่งกด
3. หมวดล่าสุดใน conversation state
4. Hybrid Router เมื่อยังจำแนกไม่ได้

## การใช้ Offline, RAG และ LLM

- ข้อความต้อนรับ เมนู การตอบรับ และคำถามชี้นำ ใช้ข้อความ offline
- การกดปุ่มต้องไม่เรียก LLM Router
- คำถามสั้นต่อเนื่องต้องเติมหมวดล่าสุดก่อนค้น RAG
- คำถามซ้ำให้ตรวจ cache ก่อน Router และ RAG
- คำถามนอกขอบเขตให้ Offline Gate ตอบ
- คำถามแบบพิมพ์อิสระที่จำแนกไม่ได้จริงจึงใช้ Hybrid Router
- ถ้าไม่มีเอกสารเกี่ยวข้อง ห้ามให้ LLM เดาคำตอบ ให้ใช้ fallback ที่กำหนด

## การจัดการสถานะ

เก็บอย่างน้อย:

- category: หมวดประกันปัจจุบัน
- menu_level: ระดับเมนูปัจจุบัน
- topic: หัวข้อย่อยล่าสุด
- answer_cache: คำตอบที่เคยสร้าง
- chat_history: ประวัติข้อความ

สถานะต้องเปลี่ยนตามการกดปุ่มหรือคำถามล่าสุด ไม่ผูกผู้ใช้ไว้กับหมวดเก่า

## น้ำเสียง

- สุภาพ เป็นมิตร มั่นใจ และช่วยลดความกังวล
- หนึ่งข้อความมีหนึ่งเป้าหมาย
- ไม่ถามหลายเรื่องยาวๆ พร้อมกัน
- ไม่อ้างว่าตรวจสอบหรือดำเนินการสำเร็จ ถ้าระบบยังไม่ได้ทำจริง
- ไม่แต่งราคา โปรโมชั่น ความคุ้มครอง หรือเงื่อนไขกรมธรรม์

## เกณฑ์ตรวจรับ

- เปิดหน้าเว็บแล้วเห็นคำทักทายและเมนูทันที
- ผู้ใช้พิมพ์เองได้โดยไม่ต้องกดปุ่ม
- ผู้ใช้เปลี่ยนหมวดกลางทางได้
- คำถามสั้นใช้บริบทล่าสุดอย่างถูกต้อง
- คำถามที่ระบุหมวดใหม่ไม่ถูกบริบทเก่าทับ
- เมนูและ canned response ไม่เรียก LLM
- ทุกพฤติกรรมสำคัญมี unit test ก่อนแก้โค้ด
