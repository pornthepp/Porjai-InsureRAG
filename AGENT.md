# AGENT.md

## กติกาบังคับสำหรับระบบสนทนา

ก่อนออกแบบ แก้โค้ด หรือรีวิวพฤติกรรม chatbot ต้องอ่าน “Rule.md” ทั้งไฟล์และทำตาม

- “Rule.md” เป็นข้อกำหนดหลักด้าน conversation flow, เมนู, state, Offline/RAG/LLM และน้ำเสียง
- ถ้าโค้ดปัจจุบันขัดกับ “Rule.md” ให้เพิ่มเทสที่แสดงพฤติกรรมที่ต้องการก่อน แล้วจึงแก้โค้ด
- ถ้าคำสั่งล่าสุดของผู้ใช้ขัดกับ “Rule.md” ให้ทำตามคำสั่งล่าสุดของผู้ใช้ และอัปเดต “Rule.md” เมื่อการตัดสินใจนั้นเป็นกติกาถาวร
- ห้ามอ้างว่า “Rule.md” เป็นเอกสารภายในของ True เอกสารนี้เป็นเพียงรูปแบบที่สกัดจากตัวอย่างการใช้งานของผู้ใช้
## เป้าหมายของโปรเจกต์

อ่านไฟล์ 2026-06-30-hybrid-router และยึดตามการทำงานตามนั้น

โปรเจกต์นี้ไม่ควรเป็นแค่ chatbot ที่ตอบได้ทุกอย่าง แต่ควรเป็น RAG ที่ตอบเฉพาะเรื่องที่มีข้อมูลจริง และไม่เรียก LLM เกินจำเป็น

เป้าหมายหลักมี 4 ข้อ

1. ตอบเรื่องประกันรถยนต์และประกันชีวิตได้จากข้อมูลในระบบ
2. ไม่ตอบมั่ว ถ้าข้อมูลไม่พอให้ถามกลับหรือบอกตรง ๆ
3. ลดค่าใช้จ่ายจากการเรียก LLM ให้มากที่สุด
4. ทำให้ flow การตอบดูเป็นธรรมชาติสำหรับลูกค้า

## RAG ที่ดีสำหรับโปรเจกต์นี้ควรเป็นแบบไหน

RAG ที่ดีของโปรเจกต์นี้ต้องมีลำดับคิดแบบนี้

1. กรองก่อน
2. จัดหมวดก่อน
3. ค้นข้อมูลเมื่อจำเป็น
4. ให้ LLM ตอบเฉพาะตอนมีข้อมูลพอ
5. ถ้าข้อมูลไม่พอ ห้ามเดา

สรุปสั้น ๆ:

- ถ้าเป็นคำทักทาย หรือคุยนอกเรื่อง ให้ตอบ offline
- ถ้าเป็นคำถามกว้างเกินไป ให้ถามกลับก่อน
- ถ้าเป็นคำถามในโดเมนและชัดเจน ค่อยเข้า retrieval
- ถ้าค้นแล้วไม่เจอหรือเจอไกลเกินไป ให้หยุด ไม่ฝืนตอบ
- ถ้าคำถามซ้ำ ให้ใช้ cache

## หลักการสำคัญ

### 1. อย่าเรียก LLM ถ้ายังไม่จำเป็น

LLM เป็นของแพงสุดในระบบนี้ เพราะงั้นทุกชั้นควรถามก่อนว่า

- ตอบแบบ rule-based ได้ไหม
- ถามกลับสั้น ๆ ได้ไหม
- ใช้ cache ได้ไหม
- ข้าม step นี้ได้ไหม

ถ้าคำตอบคือได้ ให้ทำแบบนั้นก่อน

### 2. คำถามกว้าง ๆ ต้องไม่วิ่งเข้า RAG ตรง ๆ

ตัวอย่างคำถามกว้าง

- `ประกัน`
- `ประกันชีวิต`
- `ประกันรถ`
- `ประกันรถยนต์`
- `มีแบบไหนบ้าง`

คำถามพวกนี้ไม่ควรเข้า retrieval ทันที
ควรถามกลับก่อน เช่น

- สนใจเรื่องความคุ้มครอง ราคา หรือการเคลม
- สนใจประกันรถยนต์หรือประกันชีวิต

นี่คือชั้น `clarification layer` ซึ่งสำคัญมาก

### 3. Retrieval ต้องยอมแพ้เป็น

ถ้าค้นแล้ว

- ไม่เจอเอกสาร
- เจอแต่ระยะไกลเกินไป
- เจอเอกสารแต่ไม่ตรงคำถาม

ระบบควรหยุด แล้วตอบแบบ fallback หรือถามกลับ
ห้ามฝืนให้ LLM แต่งคำตอบ

### 4. Answer ต้องยึดจาก context เท่านั้น

LLM มีหน้าที่เรียบเรียงคำตอบ ไม่ใช่สร้างความจริงใหม่

กฎคือ

- มี context ค่อยตอบ
- ไม่มี context ไม่ตอบมั่ว
- ถ้าบริบทไม่พอ ให้บอกว่าไม่พบข้อมูลส่วนนั้น

### 5. UX ต้องดูเหมือนคนช่วยจริง ไม่ใช่บอทที่เดา

ลูกค้าไม่สนว่า backend ซับซ้อนแค่ไหน
ลูกค้าสนแค่ว่า

- ถามแล้วเข้าใจไหม
- ถ้าถามกว้าง ระบบช่วยถามกลับไหม
- ถ้าไม่มีข้อมูล ระบบบอกตรงไหม

RAG ที่ดีจึงต้อง “คุยเป็น” พอ ๆ กับ “ค้นเป็น”

## สภาพปัจจุบันของโปรเจกต์

ตอนนี้โปรเจกต์มีฐานที่ดีแล้วหลายจุด

- มี `offline_gate.py` สำหรับกรองคำทักทายและคำนอกเรื่อง
- มี `query_router.py` สำหรับจัดหมวดแบบ offline
- มี `chat_service.py` ที่กันคำถามหมวด `general` ไม่ให้เข้า graph
- มี cache คำถามซ้ำใน `chat_service.py`
- มี `query_rewriter.py` ที่ไม่เรียก LLM เมื่อคำถามสั้นเกินไป
- มี `retrieval_pipeline.py` ที่ไม่เรียก grader ถ้าไม่เจอผลค้นหา หรือผลไกลเกินไป
- มี `retrieval_grader.py` แบบ batch เพื่อลดจำนวนรอบเรียก LLM
- มี `generator.py` ที่ไม่เรียก LLM ถ้าไม่มี documents
- มี test หลายชุดครอบส่วนสำคัญแล้ว

สรุปคือ ตอนนี้ระบบ “เริ่มประหยัดขึ้นแล้ว” แต่ยังไม่ถือว่าสมบูรณ์ในเชิง RAG ที่ดี

## ภาพรวม flow ที่ควรยึด

ควรคิด flow ของระบบแบบนี้

1. รับข้อความจากผู้ใช้
2. normalize ข้อความ
3. offline gate
4. vague question check
5. offline router
6. cache check
7. retrieval
8. relevance grading
9. answer generation
10. groundedness / safety check
11. fallback หรือ clarification ถ้าจำเป็น

ถ้าข้อไหนตอบได้ก่อน ให้จบก่อน
อย่าดันทุกข้อความไปจนถึง LLM เสมอ

## สิ่งที่โปรเจกต์ควรทำต่อ

### ระยะ 1: ปิดรูรั่วค่าใช้จ่ายก่อน

นี่คือ priority สูงสุด

1. เพิ่มชั้นถามกลับสำหรับคำถามกว้างในโดเมน
   เช่น `ประกันชีวิต`, `ประกันรถยนต์`, `มีแบบไหนบ้าง`

2. แยกคำถามกว้างเป็นแบบเฉพาะหมวด
   ตัวอย่าง:
   - `ประกันชีวิต` -> ถามกลับเรื่องความคุ้มครอง เบี้ย การเคลม ผู้รับประโยชน์
   - `ประกันรถยนต์` -> ถามกลับเรื่องความคุ้มครอง ราคา การเคลม โปรโมชั่น

3. เพิ่ม cache ให้ครอบมากขึ้น
   ตอนนี้ cache ใช้กับคำถามซ้ำตรง ๆ แล้ว
   อนาคตอาจเพิ่ม cache หลัง normalize เพื่อกันคำสะกดหรือเว้นวรรคต่างกัน

### ระยะ 2: ทำ retrieval ให้แม่นขึ้น

1. ทบทวนชุดข้อมูลใน `data/`
   ถ้าหัวข้อที่ลูกค้าถามบ่อยไม่มีในเอกสาร retrieval ก็จะล้มบ่อย

2. จัดเอกสารให้ตอบเป็นหัวข้อชัด ๆ
   เช่น
   - ความคุ้มครอง
   - ข้อยกเว้น
   - การเคลม
   - เอกสารที่ใช้
   - โปรโมชั่น
   - ราคา หรือเงื่อนไขราคา

3. ปรับการ chunk ข้อมูล
   chunk ที่ยาวเกินไปจะค้นไม่แม่น
   chunk ที่สั้นเกินไปจะขาดบริบท

4. ทบทวน threshold ของระยะค้นหา
   ถ้าเข้มเกินไปจะไม่เจออะไร
   ถ้าหลวมเกินไปจะได้เอกสารมั่ว

### ระยะ 3: ทำให้คำตอบดีขึ้นโดยไม่แพงขึ้น

1. ให้ generator ใช้ prompt ที่สั้นและชัด
2. บังคับให้ตอบจาก context เท่านั้น
3. แยก fallback ตามสถานการณ์
   เช่น
   - นอกเรื่อง
   - คำถามกว้าง
   - อยู่ในเรื่องแต่ข้อมูลไม่พอ

4. ทำ canned response (ข้อความเตรียมไว้ล่วงหน้า) ให้มากขึ้น
   ยิ่งตอบแบบ fixed ได้มาก ยิ่งประหยัด

## สิ่งที่ไม่ควรทำ

1. อย่าใช้ LLM router ถ้า keyword/router แบบ offline ยังเอาอยู่
2. อย่า rewrite ทุกคำถาม
3. อย่า grade ทุก document ทีละชิ้นถ้าทำ batch ได้
4. อย่า generate คำตอบเมื่อไม่มี context
5. อย่าใช้ fallback เดียวกับทุกสถานการณ์
6. อย่ารีบเปลี่ยน model ถ้ายังไม่ได้แก้ flow

## คำตอบตรง ๆ เรื่อง model กับ data

ถ้าระบบยังตอบมั่วหรือเรียก LLM บ่อยเกินไป ปัญหาส่วนใหญ่ไม่ได้มาจาก model ก่อน
มักมาจาก flow และกติกาในระบบ

ลำดับคิดที่ควรใช้คือ

1. แก้ gate
2. แก้ clarification
3. แก้ retrieval
4. เติม data
5. ค่อยพิจารณา model

พูดง่าย ๆ คือ
`flow สำคัญกว่า model`

## Definition of Done ของโปรเจกต์นี้

โปรเจกต์นี้จะถือว่าเข้าใกล้ RAG ที่ดีเมื่อ

- คำทักทายและคำนอกเรื่องไม่เรียก LLM
- คำถามกว้างไม่เข้า retrieval ตรง ๆ
- คำถามชัดเจนเท่านั้นที่เข้า graph
- ไม่มี context แล้วไม่ generate
- คำถามซ้ำไม่เรียก graph ซ้ำ
- retrieval ที่ไกลเกินไปไม่ถูกส่งต่อ
- คำตอบไม่หลุดจากข้อมูลในระบบ
- test ครอบจุดตัดสินใจสำคัญทุกชั้น

## Test ที่ควรมีเพิ่ม

ควรมี test สำหรับกรณีเหล่านี้เพิ่ม

1. คำถามกว้างในโดเมนไม่เข้า graph
   - `ประกันชีวิต`
   - `ประกันรถยนต์`

2. คำถามกว้างแต่มี typo แล้วยังถูกถามกลับถูกทาง
   - `ปะกันชีวิต`
   - `รถยน`

3. คำถามซ้ำหลัง normalize แล้วยังใช้ cache ได้

4. fallback แต่ละแบบตอบถูกสถานการณ์

5. มีเอกสารแต่ไม่ตรงคำถาม ต้องไม่ฝืน generate

## ลำดับงานแนะนำจากจุดนี้

ถ้าจะพัฒนาต่อ ให้ทำตามลำดับนี้

1. เพิ่ม clarification response แยกสำหรับ `ประกันชีวิต` และ `ประกันรถยนต์`
2. เพิ่ม test สำหรับ broad in-domain questions
3. ทบทวน `data/life_insurance.txt` และ `data/car_insurance.txt` ว่าครอบคำถามจริงพอหรือยัง
4. ปรับ retrieval threshold จากผลทดสอบจริง
5. แยก fallback response ให้ชัดตามสถานการณ์
6. ค่อยประเมินอีกทีว่าจำเป็นต้องเปลี่ยน model หรือไม่

## กติกาการแก้โปรเจกต์นี้ต่อไป

ทุกครั้งที่จะเพิ่ม feature หรือแก้ logic ให้ยึดกติกานี้

1. ถามก่อนว่าแก้จุดนี้จะช่วยลด LLM call ไหม
2. ถ้าไม่ช่วยลด cost หรือเพิ่มความแม่นชัดเจนจริง ๆ ให้คิดก่อนว่าควรทำไหม
3. เพิ่ม test ก่อนหรืออย่างน้อยเพิ่ม test พร้อมกัน
4. แก้ทีละชั้น อย่าแก้หลายจุดพร้อมกันจนตามสาเหตุไม่ทัน
5. ทดสอบจากพฤติกรรมผู้ใช้จริง ไม่ใช่ดูแค่ unit test ผ่าน

## สรุปสั้นที่สุด

RAG ที่ดีของโปรเจกต์นี้คือ

- กรองเก่ง
- ถามกลับเป็น
- ค้นเฉพาะตอนควรค้น
- ตอบจากข้อมูลจริง
- ไม่มั่ว
- ไม่แพง

ถ้าต้องเลือกระหว่าง “ตอบได้ทุกอย่าง” กับ “ตอบเฉพาะตอนมั่นใจ”
ให้เลือกแบบหลัง
เพราะสำหรับระบบประกันภัย ความน่าเชื่อถือสำคัญกว่าการดูฉลาด

## กติกาล่าสุดที่อนุมัติแล้ว: Hybrid Router

กติกาส่วนนี้แทนลำดับ router และ cache เดิมในเอกสาร ตั้งแต่วันที่ 2026-06-30

### Flow ที่ต้องทำตาม

1. รับและ normalize ข้อความ
2. ใช้ Offline Gate ปิดคำทักทายและคำถามนอกเรื่อง
3. คำถามกว้าง เช่น ประกัน, รถยนต์, ประกันชีวิต ให้ถามรายละเอียดเพิ่มโดยไม่เรียก LLM
4. ตรวจ cache ก่อนเรียก router เพื่อไม่จำแนกคำถามเดิมซ้ำ
5. ใช้ Offline Router ก่อนเสมอ
6. ถ้า Offline Router จำแนกได้แล้ว ห้ามเรียก LLM Router
7. ถ้าคำถามผ่าน Offline Gate แต่ Offline Router ยังคืน general จึงเรียก LLM Router ได้หนึ่งครั้ง
8. ถ้า LLM Router คืน car_insurance หรือ life_insurance จึงเข้า retrieval
9. ถ้า LLM Router คืน general, error, timeout หรือตอบผิดรูปแบบ ให้ถามรายละเอียดเพิ่มและไม่เข้า RAG
10. ถ้า retrieval ไม่มีเอกสารที่เกี่ยวข้อง ห้ามเรียก generator เพื่อแต่งคำตอบ

### ขอบเขตของ LLM Router

- คืนได้เพียง car_insurance, life_insurance หรือ general
- ใช้ temperature=0 และจำกัด token ให้น้อย
- มีหน้าที่จำแนกหมวดเท่านั้น ห้ามตอบลูกค้า
- คำถามหนึ่งครั้งเรียกได้ไม่เกินหนึ่งรอบ
- API มีปัญหาต้อง fallback เป็น general อย่างปลอดภัย

### ตัวอย่างพฤติกรรมที่ต้องได้

- ประกันรถยนต์ชั้นหนึ่งคุ้มครองอะไรบ้าง -> Offline Router -> RAG โดยไม่เรียก LLM Router
- ประกันชีวิต -> clarification โดยไม่เรียก LLM และไม่เข้า RAG
- ประกัน 2+ กับ 3+ ต่างกันยังไง -> LLM Router หนึ่งครั้ง -> car RAG
- ซ่อมห้างกับซ่อมอู่ต่างกันยังไง -> Offline Gate ต้องปล่อยผ่าน -> car RAG
- คำถามเดิมซ้ำ -> cache โดยไม่เรียก router และ graph ซ้ำ

### ลำดับงานปัจจุบัน

1. เพิ่ม failing test ให้ ซ่อมห้าง และ ซ่อมอู่ ผ่าน Offline Gate
2. เพิ่มเทส Hybrid Router ก่อนแก้ production code
3. แยก Offline Router และ LLM fallback ให้ทดสอบคนละส่วน
4. ย้าย cache lookup ไปก่อน router
